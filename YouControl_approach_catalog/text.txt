v1 — Початковий парсер (каталог по КВЕДам)

Що робив: парсив каталог списків компаній (публічна частина).

Метрика: ~4.0 с/компанія → ≈100 компаній за 7 хв.

Проблеми: Cloudflare/бот-блоки (неавторизований), сайт нестабільний (особливо 12:00–20:00).

Висновок: підходить як PoC, але непридатний для масового збору.

v2 — Асинхрон + багатопоточність

Що робив: перевів на async + threads для пришвидшення.

Метрика: ~1.6 компаній/сек → ≈100 компаній/хв.

Проблеми: агресивні захисти Юконтрол, зміна UA/паузи — не допомогли.

Висновок: швидкість зросла, але блокування лишилися критичною перешкодою.

v3 — Контакт з YouControl (людський фактор)

Що робив: дзвінки в офіс YouControl для уточнення опцій.

Відкрито: два варіанти — офіційне API або авторизація на сайті.

Висновок: треба оцінити вартість API та обмеження авторизації.

v4 — Аналіз API (вартість)

Що робив: прорахунок використання офіційного API.

Вартість: 0.8 грн / запит × 8 000 000 ≈ 6 400 000 грн.

Висновок: економічно нерелевантно — опція відхилена.

v5 — Авторизація на сайті

Що робив: логін на сайті, парсинг доступних сторінок для авторизованих.

Метрика: ~10 компаній / 0.48 с (дуже швидко при локальному запиті).

Проблеми: каталог став недоступним/змінена структура → перепис парсера; обмеження для авторизованих: 25 запитів/день або платні пакети (1500 грн за +25).

Висновок: авторизація дає швидкість, але бізнес-обмеження зводять це нанівець.

v6 — Повернення до неавторизованого парсингу з ідеєю ротації IP

Що робив: план — отримувати дані як неавторизований, але з ротацією IP.

Мета: уникнути блокувань без витрат на API / підписки.

Статус: в процесі розробки (поточні питання — якість проксі, health-check, throttle).

v7 — «Працююча, повільна» стабільна версія (тимчасова)

Що робив: повернувся до v1-логіки + більші паузи, симуляція поведінки людини, чекпоінти.

Результат: стабільний збір малими об’ємами; збереження прогресу.

Висновок: дає видимі результати зараз — потрібен масштабувальний план.
